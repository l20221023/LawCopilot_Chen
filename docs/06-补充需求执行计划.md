# LawCopilot 补充需求执行计划

## 1. 需求背景

补充文档新增了一条核心能力：`Token 成本优化引擎`。

目标不是改变当前“用户按次数扣费”的产品规则，而是在系统调用 AI 供应商时，增加一层内部成本决策：

- 估算本次请求的 token 成本
- 对比“按 token 计费”和“按次计费”两种供应商成本
- 自动选择更便宜的调用策略
- 对固定不变的 `system prompt` 引入 prompt cache，避免重复计费

这条需求是平台内部优化，不直接改变当前用户看到的计费方式。

## 2. 对当前项目的影响范围

当前项目现状：

- 用户侧计费链路已经存在：发送前额度校验，成功后写 `usage_logs`，再扣减次数
- AI 调用链已经存在：`Browser -> /api/chat -> OpenRouter`
- token 记录已经存在：`usage_logs`
- 场景 system prompt 已经存在：来自场景配置

当前缺口：

- 还没有独立的“成本路由模块”
- token 估算逻辑还只是发送完成后的简单记录，不参与发送前决策
- 后端 `/api/chat` 还没有多计费策略输入
- system prompt 没有缓存命中判断
- usage 记录里没有保存“本次选择了哪种成本策略”

## 3. 需求拆解

### 3.1 成本路由决策

新增一个独立模块，负责根据当前请求上下文估算成本并选策略。

输入：

- `scenarioId`
- `systemPrompt`
- 历史消息
- 当前用户消息
- 附件提取文本
- token 单价配置
- 按次固定价格配置
- 是否命中 prompt cache

输出：

- `strategy`: `token` | `fixed`
- `estimated_input_tokens`
- `estimated_output_tokens`
- `estimated_total_tokens`
- `estimated_token_cost`
- `estimated_fixed_cost`
- `cached_system_prompt`
- `system_prompt_token_count`

### 3.2 Prompt Cache

对同一会话内固定的 `system prompt` 做哈希计算。

规则：

- 同一个会话内，如果当前 `system prompt hash` 和上一次一致，则视为命中缓存
- 命中缓存时：
  - 成本估算时不再把 system prompt token 计入输入成本
  - 向后端请求中附带 `cached_system_prompt: true`
- 未命中缓存时：
  - 正常计算 system prompt token
  - 更新当前会话缓存状态

### 3.3 Usage 扩展

当前 `usage_logs` 只记录实际 token 和 credit_cost。

建议新增字段：

- `billing_strategy`
- `estimated_input_tokens`
- `estimated_output_tokens`
- `estimated_total_tokens`
- `estimated_token_cost`
- `estimated_fixed_cost`
- `cached_system_prompt`
- `system_prompt_hash`

如果本轮不想动数据库，也可以先把这些信息写入 `metadata jsonb`。

## 5. 建议的模块设计

### 5.1 新增模块

建议新增：

- `web/src/lib/ai/cost-strategy-engine.ts`
- `web/src/lib/ai/token-estimator.ts`
- `web/src/lib/ai/prompt-cache.ts`

职责：

- `token-estimator.ts`
  - 粗略估算 token
  - 支持文本、消息数组、附件文本
- `prompt-cache.ts`
  - 计算 hash
  - 判断当前会话是否命中缓存
- `cost-strategy-engine.ts`
  - 输入请求上下文
  - 输出路由决策结果

### 5.2 类型设计

建议新增类型：

```ts
type BillingStrategy = 'token' | 'fixed'

type CostStrategyDecision = {
  strategy: BillingStrategy
  estimatedInputTokens: number
  estimatedOutputTokens: number
  estimatedTotalTokens: number
  estimatedTokenCost: number
  estimatedFixedCost: number
  cachedSystemPrompt: boolean
  systemPromptHash: string
}
```

### 5.3 配置项

建议先做成本常量配置，不直接写死在业务函数里：

- `TOKEN_PRICE_PER_1K`
- `FIXED_CALL_PRICE`
- `DEFAULT_ESTIMATED_OUTPUT_TOKENS`

放置位置建议：

- `web/src/lib/ai/cost-config.ts`

## 6. 执行阶段计划

### Phase 1：建立基础决策能力

目标：

- 完成 token 估算
- 完成成本对比
- 能输出策略选择结果

任务：

- 新增 `token-estimator.ts`
- 新增 `cost-config.ts`
- 新增 `cost-strategy-engine.ts`
- 为纯文本、历史消息、附件文本写基础单元验证

完成标准：

- 能给任意一组消息内容输出稳定决策结果

### Phase 2：接入 prompt cache

目标：

- 在会话维度识别 system prompt 是否重复
- 命中缓存时，不再把 system prompt 计入预估输入 token

任务：

- 新增 `prompt-cache.ts`
- 在会话请求构建阶段加入 `systemPromptHash`
- 在 `use-chat-controller.ts` 发送前接入缓存判断

完成标准：

- 同一会话连续发送两条消息时，第二次如果场景未变，则 `cached_system_prompt = true`

### Phase 3：接入真实发送链路

目标：

- 让 `/api/chat` 收到策略决策结果
- 后端能够根据策略执行不同调用分支

任务：

- 扩展 `buildChatApiPayload`
- 扩展 `streamChatCompletion`
- 扩展 `web/api/chat.js`
- 先保留同一个 OpenRouter 入口，但逻辑上支持按策略分支

完成标准：

- 后端日志或返回结果能确认已收到 `billing_strategy` 和 `cached_system_prompt`

### Phase 4：补 usage 与可观测性

目标：

- 记录“为何选这个策略”
- 为后续成本分析留数据

任务：

- 扩展 `UsageRecordInput`
- 扩展 `usage_logs`
- 在 `recordUsageLog()` 中写入策略结果

完成标准：

- 能回溯每次请求的预估成本、实际 token、是否命中缓存

## 7. 推荐开发顺序

按当前项目状态，建议这样执行：

1. 先做 `token-estimator.ts`
2. 再做 `cost-strategy-engine.ts`
3. 再做 `prompt-cache.ts`
4. 然后把这三者接入 `use-chat-controller.ts`
5. 再扩展 `/api/chat`
6. 最后补 `usage_logs`

原因：

- 先把纯计算模块做出来，风险最小
- 再改发送链路，避免一上来就在流式请求里混业务判断
- usage 扩展放最后，避免数据库改动阻断主逻辑验证

## 8. 验收标准

补充需求完成后，至少应满足：

- 同一请求发送前，系统能输出一个明确的成本策略
- 当上下文较短时，优先选择 `token`
- 当上下文较长时，优先选择 `fixed`
- 同一会话内连续发送，如果 system prompt 未变化，则命中缓存
- usage 记录里能追踪本次策略决策结果
- 原有用户扣次逻辑不受影响
- AI 流式输出链路不被破坏

## 9. 风险与注意事项

### 9.1 不要把策略逻辑硬编码进发送函数

这是这份补充需求最重要的工程要求。`use-chat-controller.ts` 已经很长，再继续直接堆逻辑，后面不可维护。

### 9.2 token 估算不要求精确，但必须稳定

这次需求考的是“有估算意识”，不是做 tokenizer 级别精确统计。当前项目可以接受粗略估算。

### 9.3 Prompt Cache 先做本地会话级，不要一上来做跨会话全局缓存

否则复杂度会立刻上升，且当前收益不高。

### 9.4 用户计费与供应商成本要分开

当前用户仍是“按次数扣费”。补充需求优化的是平台成本，不是用户可见计费规则。

## 10. 当前建议

基于现有项目状态，下一步最合适的是：

1. 先实现 `Phase 1 + Phase 2`
2. 本地验证成本策略和 prompt cache 判断
3. 再决定是否同步扩展 `usage_logs` 表结构

如果要继续开发，建议直接进入下面这个具体开发包：

- `P1 TokenEstimator`
- `P2 CostStrategyEngine`
- `P3 PromptCache`
- `P4 SendFlow Integration`
- `P5 Usage Schema Extension`
