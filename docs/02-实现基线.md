# LawCopilot 实现基线

## 1. 目录建议

```text
src/
  app/
    router.tsx
    providers.tsx
  components/
    layout/
    chat/
    billing/
    settings/
    common/
  features/
    auth/
    chat/
    scenarios/
    billing/
    settings/
    usage/
  lib/
    supabase/
    ai/
    utils/
  pages/
    auth/
    chat/
    pricing/
    settings/
  types/
  constants/
  hooks/
```

原则：

- 页面负责装配，不承载复杂业务
- 业务逻辑放进 `features`
- 第三方接入放进 `lib`
- 类型集中管理

## 2. 数据模型建议

```ts
type Conversation = {
  id: string;

  created_at: string;
  updated_at: string;
```

### 2.3 messages

```ts
type AttachmentType = "image" | "pdf" | "txt";

type MessageAttachment = {
```

### 2.4 usage_logs

```ts
type UsageLog = {
```

## 3. 最小数据库表建议

建议最少四张表：

- `users`
- `conversations`
- `messages`
- `usage_logs`

如果附件要持久化到存储桶，可增加：

- `attachments` 独立表

但首版也可以先把附件信息保存在 `messages.attachments` JSON 字段中。

## 5. 状态划分建议

### 5.1 全局状态

- 当前登录用户
- 用户资料
- 当前额度与订阅状态

### 5.2 页面级状态

- 当前选中的会话
- 当前输入文本
- 当前上传附件列表
- 当前生成状态

### 5.3 派生状态

- 是否可以发送
- 是否显示额度警告
- 当前订阅是否过期

## 6. 发送消息的调用链

建议固定为如下顺序：

1. 前端校验输入是否为空
2. 校验用户额度或订阅状态
3. 保存用户消息草稿
4. 读取会话历史
5. 注入场景 `system_prompt`
6. 解析附件文本
7. 发起流式 AI 请求
8. 持续更新 assistant 消息
9. 请求完成后记录 token
10. 对按次用户执行扣费
11. 刷新用户资料与会话更新时间

## 8. OpenRouter 请求建议

请求建议不要直接从浏览器调用第三方模型，而是走服务端中转或 Supabase Edge Function。

最小输入结构：

```ts
type AIRequestPayload = {
  model: string;
  messages: Array<{
    role: "system" | "user" | "assistant";
    content: string | Array<Record<string, unknown>>;
  }>;
  stream: boolean;
};
```

前端只关心：

- 流式文本增量
- 停止按钮
- 错误提示
- token 用量回传

## 9. UI 基线

建议做成简洁商务风，不走炫技路线。

- 主色：深蓝或墨绿
- 背景：浅灰白
- 强调色：琥珀或青蓝
- 圆角和阴影适度
- 信息分区清晰

主对话页建议分区：

- 左侧：会话列表 + 新建对话 + 额度状态
- 右侧顶部：场景选择 + 会话标题
- 中部：消息流
- 底部：输入框 + 上传按钮 + 发送按钮

## 10. 开发顺序建议

### 阶段 1

- 初始化项目
- 接入 Tailwind

- 接入 Supabase
- 写基础 layout

### 阶段 2

- 做登录注册
- 做用户资料加载
- 写路由守卫

### 阶段 3

- 写场景数据
- 写主聊天页面
- 做会话列表和消息列表

### 阶段 4

- 接流式 AI
- 做额度校验
- 做扣费和 usage 记录

### 阶段 5

- 定价页
- 设置页
- 联调和验收
